06

Flujos críticos

Recorridos principales a través de las distintas capas del sistema

📅 Reserva de cita (paciente)

1
patient-portal → GET /api/v1/slots?service=&center=&date=
2
Scheduling → consulta availability_rules, calcula huecos libres (slot_allocations)
3
patient-portal → POST /api/v1/appointments {slot_id, idempotency_key}
4
Scheduling → transacción: INSERT slot_allocation + INSERT appointment + evento AppointmentBooked
5
Audit → escucha AppointmentBooked → escribe audit_event con hash chain
6
Notifications (futuro) → outbox → email/SMS de confirmación

📄 Descarga de documento (paciente)

1
patient-portal → GET /api/v1/documents → lista publicaciones activas
2
Documents → verifica patient_id + publicación activa (withdrawn_at IS NULL)
3
patient-portal → POST /api/v1/documents/{id}/download-auth
4
Documents → crea document_download_grant (token_hash, expires_at=5min)
5
patient-portal → GET /api/v1/documents/download/{grant_token}
6
Documents → consume grant (marca consumed_at) → stream desde Object Storage → audit

🔐 Autenticación BFF (Sanctum cookie)

1
portal → GET /api/v1/csrf-cookie → recibe XSRF-TOKEN
2
portal → POST /api/v1/login {email, password, _csrf}
3
Identity → verifica credenciales → crea session (Redis) → Set-Cookie HttpOnly
4
Audit → login event con IP, dispositivo, ubicación
5
auth lib → sessionExpiryInterceptor renueva automáticamente antes de expirar

🔄 Cancelación de cita (versionada)

1
portal → POST /api/v1/appointments/{id}/cancel {reason_code, idempotency_key}
2
Scheduling → verifica ChangePolicy (ventana de cancelación, estado)
3
Scheduling → transacción: version++, status=cancelled, DELETE slot_allocation, INSERT appointment_change
4
Audit → AppointmentCancelled event → hash chain record
5
Notifications (futuro) → slot liberado → oferta a lista de espera
